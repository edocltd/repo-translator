# repo-translator — Аналіз проблем та захисних механізмів

Кожна проблема має: опис, як виявили (або теоретичний сценарій), ризик, та конкретний захист.

---

## Категорія 1: Сканування та аналіз репозиторію

### P1.1: Гігантський репозиторій

**Сценарій**: Користувач запускає інструмент на монорепо з 50 000 файлів (Linux kernel, Chromium).

**Ризик**: Сканування зависає, пам'ять переповнюється.

**Захист**:
- Ліміт глибини сканування (default: 10 рівнів)
- Ліміт кількості файлів для перекладу (default: 500 .md файлів)
- При перевищенні — попередження + пропозиція звузити scope через `include`
- Показувати progress bar при скануванні

```
⚠️ Знайдено 12,847 .md файлів. Максимум за замовчуванням: 500.
Рекомендація: звузьте область через include у .repo-translator.yaml
Або запустіть з --max-files 12847 щоб зняти ліміт.
```

### P1.2: Не-git репозиторій

**Сценарій**: Папка без `.git/` — просто набір файлів.

**Ризик**: Неможливо відстежувати SHA для синхронізації. `sync-check` не працюватиме.

**Захист**:
- При `init` перевіряти наявність `.git/`
- Якщо немає — попередження: "Синхронізація змін не працюватиме без git"
- Замість SHA використовувати hash файлу (md5/sha256) для відстеження змін
- Інструмент ПРАЦЮЄ без git, просто без sync

### P1.3: Репо з існуючою структурою перекладів

**Сценарій**: Проєкт вже має `translations/`, `i18n/`, `uk/`, `lang/` з перекладами.

**Ризик**: Інструмент затре існуючі переклади або створить конфліктну структуру.

**Захист**:
- При `init` сканувати на наявність типових i18n-каталогів
- Якщо знайдено — запитати:

```
Знайдено існуючу структуру перекладів:
  uk/ (12 файлів)
  vi/ (8 файлів)

Варіанти:
  1. Імпортувати в translations/ (перемістити існуючі)
  2. Працювати з існуючою структурою (translations_dir: .)
  3. Створити нову паралельну структуру
  4. Скасувати
```

### P1.4: Symlinks та circular references

**Сценарій**: Репо містить символічні посилання що вказують на батьківський каталог → нескінченний цикл.

**Ризик**: Сканер зависає або падає.

**Захист**:
- Не слідувати за symlinks при скануванні (default)
- Відстежувати вже відвідані каталоги (set of realpath)
- Ліміт глибини рекурсії

### P1.5: Бінарні файли з розширенням .md

**Сценарій**: Файл `data.md` який насправді бінарний (згенерований автоматично, містить null-байти).

**Ризик**: Інструмент намагається "перекласти" бінарні дані.

**Захист**:
- Перед обробкою перевірити: чи файл текстовий?
  - Прочитати перші 8 KB, шукати null-байти (0x00)
  - Якщо > 1% null-байтів — класифікувати як бінарний, пропустити
- Логувати: "Пропущено data.md — бінарний файл"

### P1.6: Порожні .md файли

**Сценарій**: Файл існує але порожній (0 байтів) або містить тільки frontmatter.

**Ризик**: AI отримує порожній промпт, повертає галюцинацію або помилку.

**Захист**:
- Якщо файл < 10 байтів — пропустити (логувати)
- Якщо файл містить тільки frontmatter без прози — скопіювати as-is (не перекладати)

### P1.7: Файли з однаковими іменами в різних каталогах

**Сценарій**: 10 файлів README.md у різних підкаталогах. При flat-режимі роботи — конфлікт імен.

**Ризик**: Файли перезаписують одне одного.

**Захист**:
- Завжди зберігати повну структуру каталогів:
  `translations/uk/docs/README.md` та `translations/uk/api/README.md` — різні файли
- Ніколи не робити flat-режим

## Категорія 2: Переклад — якість AI-виводу

### P2.1: AI скорочує файл (РЕАЛЬНА, claude-howto)

**Сценарій**: Оригінал 1945 рядків, AI повернув 540 (втрачено 58%).

**Як виявили**: Порівняння розмірів файлів. Кирилічний переклад має бути ~40% БІЛЬШИЙ за оригінал, а не менший.

**Захист**:
- Після перекладу ОБОВ'ЯЗКОВО перевіряти:

```python
ratio = translated_lines / original_lines
if ratio < 0.85:
    ERROR(f"Файл скорочено: {ratio:.0%} від оригіналу. "
          f"Очікується ≥95%. Переклад відхилено.")
if ratio > 2.0:
    WARNING(f"Файл подвоївся: {ratio:.0%}. Можливо AI додав зайвий контент.")
```

- При помилці — не зберігати файл, повідомити користувача
- Зберігати оригінал і зламаний переклад поруч для порівняння

### P2.2: AI перекладає код всередині блоків (РЕАЛЬНА)

**Сценарій**: `mkdir -p .claude/commands` перекладається як `створити_каталог .claude/commands`.

**Захист**:
- ПЕРЕД відправкою AI — витягти всі код-блоки та замінити на плейсхолдери
- ПІСЛЯ отримання перекладу — повернути оригінальні код-блоки

```
Оригінал:
  Текст проза
  ```bash
  mkdir -p .claude
  ```
  Ще текст

Відправляємо AI:
  Текст проза
  %%%CODE_BLOCK_1%%%
  Ще текст

Отримуємо від AI:
  Перекладений текст
  %%%CODE_BLOCK_1%%%
  Ще перекладений текст

Збираємо назад:
  Перекладений текст
  ```bash
  mkdir -p .claude
  ```
  Ще перекладений текст
```

**Ключове**: Код НІКОЛИ не проходить через AI. Тільки проза.

### P2.3: AI перекладає Mermaid-мітки

**Сценарій**: `A["Main Agent"]` → `A["Головний агент"]` — ламає діаграму якщо інші вузли посилаються на "Main Agent".

**Захист**:
- Mermaid-блоки = код-блоки. Обробляються як P2.2 — витягуються та повертаються без змін.

### P2.4: AI змінює Markdown-форматування

**Сценарій**: Оригінал `**bold text**`, AI повертає `__bold text__` або взагалі прибирає форматування.

**Захист**:
- У промпті: чітка інструкція "зберігати Markdown-форматування 1:1"
- Валідація: порівняти кількість `**`, `*`, `#`, `|`, `-` в оригіналі та перекладі
  - Якщо кількість `#` заголовків не збігається — ПОМИЛКА
  - Якщо кількість `|` (таблиці) різниться значно — ПОПЕРЕДЖЕННЯ

### P2.5: AI "галюцинує" URL

**Сценарій**: Оригінал `[docs](https://docs.example.com)`, AI змінює на `[документація](https://docs.example.com/uk)` — URL не існує.

**Захист**:
- Витягти всі URL з оригіналу
- Витягти всі URL з перекладу
- Порівняти множини: якщо URL додано/видалено/змінено — ПОПЕРЕДЖЕННЯ

### P2.6: AI видаляє HTML-теги

**Сценарій**: `<picture><source srcset="...">` зникає з перекладу.

**Захист**:
- HTML-теги = код. Витягуються плейсхолдерами (як код-блоки)
- Після збірки — перевірити кількість HTML-тегів

### P2.7: AI додає пояснення від себе

**Сценарій**: AI вставляє "Примітка перекладача: ..." або "Переклад може бути неточним".

**Захист**:
- У промпті: "НЕ додавай пояснень, коментарів чи приміток від себе"
- Валідація: шукати типові патерни ("note:", "translator's note", "примітка:")

### P2.8: Великий файл не вміщується в контекст (РЕАЛЬНА, claude-howto)

**Сценарій**: Файл 87 KB / 3136 рядків — перевищує ліміт контексту моделі.

**Захист**:
- Розбиття по `##` заголовках (chunk by H2)
- Якщо секція > 4000 токенів — розбити по `###`
- Якщо навіть `###` секція > 4000 — розбити по порожніх рядках
- Кожен чанк отримує той самий глосарій та правила
- Після перекладу — перевірити що чанки зібрались без дублювання/пропусків

### P2.9: AI повертає неповну відповідь (обрізку)

**Сценарій**: API повертає відповідь яка обривається на середині речення (max_tokens досягнуто).

**Захист**:
- Перевірити чи останній рядок — повне речення (закінчується на `.`, `:`, або порожній рядок)
- Перевірити чи кількість ``` пар — парна (всі блоки закриті)
- Якщо обрізка — повторити запит з меншим чанком

## Категорія 3: Посилання та якорі

### P3.1: Зламані якорі після перекладу (РЕАЛЬНА, claude-howto)

**Сценарій**: `[Дивись нижче](#slash-commands)` але заголовок тепер `## Слеш-команди` → якір `#слеш-команди`.

**Як виявили**: `check_cross_references.py` — 10 помилок у concepts guide, 3 в модулях.

**Захист**:
- ПІСЛЯ перекладу автоматично:
  1. Зібрати всі заголовки → згенерувати GitHub-style якорі
  2. Знайти всі `(#anchor)` посилання
  3. Якщо anchor не існує в заголовках → знайти найближчий (fuzzy match)
  4. Замінити автоматично
  5. Логувати кожну заміну

### P3.2: Апостроф в якорях (РЕАЛЬНА, claude-howto)

**Сценарій**: Заголовок `## Пов'язані концепції`. Апостроф `'` (U+0027) видаляється при генерації якоря → `#повязані-концепції`. Але посилання містить `ʼ` (U+02BC) → `#повʼязані-концепції` → mismatch.

**Захист**:
- При генерації якорів — нормалізувати ВСІ варіанти апострофу (U+0027, U+02BC, U+2019, U+2018) до одного
- Або краще — видаляти всі апострофи при порівнянні якорів

### P3.3: Дублюючі заголовки (РЕАЛЬНА, claude-howto)

**Сценарій**: Два заголовки `## Prompt-хуки` — GitHub додає `-1` до другого (`#prompt-хуки-1`). Скрипт валідації цього не знає.

**Як виявили**: `check_cross_references.py` не підтримує GitHub-нумерацію дублікатів.

**Захист**:
- При генерації якорів — відстежувати лічильник для кожного якоря
- Другий `## Prompt-хуки` → `#prompt-хуки-1`
- Або краще: при перекладі виявляти дублюючі заголовки та робити їх унікальними

### P3.4: Відносні посилання на непереведені файли

**Сценарій**: Переведений `translations/uk/docs/guide.md` містить `[скрипт](../src/deploy.sh)`. Шлях `../src/deploy.sh` відносно translations/uk/docs/ вказує на translations/uk/src/deploy.sh — якого не існує.

**Захист**:
- При створенні перекладу — перерахувати всі відносні посилання:
  1. Посилання на інший .md що перекладений → залишити (файл поруч)
  2. Посилання на .md що НЕ перекладений → переписати на оригінал
  3. Посилання на не-.md файл → переписати на оригінал
- Формула перерахунку глибини:
  `../../` → потрібно додати стільки `../` скільки рівнів від translations/{lang}/ до кореня

### P3.5: Абсолютні шляхи

**Сценарій**: Файл містить `/docs/guide.md` (абсолютний шлях від кореня репо, типово для Hugo/Jekyll). Переклад знаходиться в `/translations/uk/docs/guide.md`.

**Захист**:
- Виявляти абсолютні шляхи при скануванні
- ПОПЕРЕДЖЕННЯ: "Файл X містить абсолютні шляхи — вони не будуть переписані автоматично"
- Для Hugo/Jekyll/Docusaurus — окремі правила (якщо буде попит)

### P3.6: Посилання з якорями на інші файли

**Сценарій**: `[Дивись](#section)` — якір в тому ж файлі. `[Дивись](other.md#section)` — якір в іншому файлі. Якщо other.md перекладений — `#section` може стати `#секція`.

**Захист**:
- При валідації cross-file якорів:
  1. Резолвити .md файл
  2. Якщо файл в translations/ — перевіряти якорі ПЕРЕКЛАДЕНОГО файлу
  3. Якщо файл в оригіналі — перевіряти якорі ОРИГІНАЛУ

## Категорія 4: Кодування та формат файлів

### P4.1: Файл не в UTF-8 (РЕАЛЬНА, claude-howto)

**Сценарій**: AI-модель повертає текст частково в CP1251, частково в UTF-8. 240 невалідних байтів.

**Як виявили**: `file` команда показала "Non-ISO extended-ASCII". Python `decode('utf-8')` кинув UnicodeDecodeError.

**Захист**:
- ПІСЛЯ КОЖНОГО запису файлу:

```python
with open(path, 'rb') as f:
    raw = f.read()
try:
    raw.decode('utf-8')
except UnicodeDecodeError as e:
    # Спроба автовиправлення
    fixed = try_fix_encoding(raw)
    if fixed:
        write(path, fixed)
        WARNING(f"Виправлено кодування: {e}")
    else:
        ERROR(f"Невалідне кодування, файл не збережено")
```

- Функція `try_fix_encoding`: знайти блок з невалідними байтами, спробувати декодувати як CP1251/Latin-1, перекодувати в UTF-8

### P4.2: BOM (Byte Order Mark)

**Сценарій**: Оригінальний файл має BOM (0xEF 0xBB 0xBF на початку). Деякі інструменти ламаються.

**Захист**:
- При читанні — ігнорувати BOM
- При записі — НЕ додавати BOM (стандарт для веб та Git)

### P4.3: CRLF vs LF

**Сценарій**: Оригінал з LF (Unix), переклад з CRLF (Windows). Git diff показує зміни в кожному рядку.

**Захист**:
- При записі — використовувати те ж закінчення рядків що в оригіналі
- Або завжди LF (Git сам конвертує через `.gitattributes`)
- ПОПЕРЕДЖЕННЯ якщо `.gitattributes` не налаштований

### P4.4: Відсутній newline в кінці файлу

**Сценарій**: Markdownlint (MD047) вимагає newline в кінці файлу.

**Захист**:
- Завжди додавати `\n` в кінці файлу
- Перевіряти при валідації

### P4.5: Invisible Unicode characters

**Сценарій**: AI вставляє zero-width spaces (U+200B), zero-width joiners (U+200D), або directional marks (U+200F) які ламають рендеринг.

**Захист**:
- Після перекладу — видалити invisible Unicode:
  U+200B, U+200C, U+200D, U+200E, U+200F, U+FEFF (крім BOM на позиції 0)
- Логувати якщо знайдено

## Категорія 5: Специфіка мов

### P5.1: RTL мови (арабська, іврит)

**Сценарій**: Переклад арабською — текст справа наліво, але код-блоки та Markdown-синтаксис — LTR.

**Захист**:
- При ініціалізації RTL мови — ПОПЕРЕДЖЕННЯ:
  "RTL мови потребують додаткової уваги. Код-блоки та таблиці можуть рендеритись некоректно."
- Додавати `dir="rtl"` якщо потрібно для HTML-файлів
- Для чистого Markdown — обмежені можливості (GitHub рендерить RTL автоматично)

### P5.2: CJK мови (китайська, японська, корейська)

**Сценарій**: В CJK немає пробілів між словами. Рядки стають КОРОТШИМИ (менше байтів на слово). Перевірка "рядки ±5%" може хибно спрацювати.

**Захист**:
- Коефіцієнт перевірки рядків залежить від мови:
  - Кирилиця (uk, ru, bg): очікувати +10-40% рядків
  - CJK (zh, ja, ko): очікувати -20-40% рядків (бо текст компактніший)
  - Латиниця (de, fr, es): очікувати ±10%
- Перевірка за кількістю рядків менш жорстка, основна перевірка — код-блоки та заголовки

### P5.3: Мови з особливими символами

**Сценарій**: Турецька мова — `I` → `ı` (без крапки), `i` → `İ` (з крапкою). Можуть ламати case-insensitive порівняння.

**Захист**:
- Використовувати Unicode-aware порівняння (`str.casefold()` замість `.lower()`)
- При генерації якорів — використовувати `.lower()` з `re.UNICODE`

### P5.4: Глосарій для конкретної мови

**Сценарій**: "commit" українською = "коміт" (транслітерація), російською = "коммит", німецькою = "Commit" (не перекладається).

**Захист**:
- Глосарій генерується ПЕР МОВУ: `translations/uk/.glossary.yaml`
- При init — генерувати базовий глосарій з поширеними термінами для обраної мови
- Вбудована база термінів для 10-15 найпопулярніших мов

## Категорія 6: Frontmatter та метадані

### P6.1: YAML frontmatter ламається при перекладі

**Сценарій**: Frontmatter `description: "Deploy to production"` AI перекладає як `description: "Розгортання на продакшен"` — OK. Але якщо description містить `:` — YAML ламається: `description: Deploy: production step` → невалідний YAML.

**Захист**:
- Frontmatter обробляти окремо від тіла файлу
- Парсити YAML → перекладати тільки значення визначених полів → серіалізувати назад
- Поля для перекладу: `description`, `title`
- Поля НЕ перекладати: `name`, `tools`, `model`, `allowed-tools`, `tags`
- Завжди обгортати значення в лапки після перекладу

### P6.2: Подвійний frontmatter

**Сценарій**: Файл з двома `---` блоками — один справжній frontmatter, інший — горизонтальна лінія в Markdown.

**Захист**:
- Frontmatter — тільки перші `---` ... `---` на початку файлу
- Все що після першого закриваючого `---` — тіло файлу

### P6.3: Tilde-огороджені блоки (`~~~~`)

**Сценарій**: Деякі .md файли (як claude-howto) використовують `~~~~` замість ` ``` ` для вкладених код-блоків.

**Захист**:
- Обробляти і ` ``` ` і `~~~~` як маркери код-блоків
- Підтримувати довільну кількість тильд (мінімум 3)

## Категорія 7: Файлова система

### P7.1: Немає прав запису

**Сценарій**: Каталог read-only або файл заблокований іншим процесом.

**Захист**:
- Перед початком — перевірити права запису в target каталозі
- При помилці запису — чітке повідомлення: "Немає прав запису в translations/uk/"

### P7.2: Диск заповнений

**Сценарій**: Переклад великого репо заповнює диск на середині процесу.

**Захист**:
- Перед початком — оцінити потрібний простір (2× розмір оригінальних .md файлів)
- Перевірити вільне місце
- При помилці запису — коректне завершення з повідомленням

### P7.3: Дуже довгі шляхи (Windows)

**Сценарій**: `translations/uk/very/deep/nested/directory/structure/README.md` — на Windows ліміт 260 символів.

**Захист**:
- При скануванні — перевіряти довжину результуючого шляху
- Якщо > 250 — ПОПЕРЕДЖЕННЯ

### P7.4: Спеціальні символи в іменах файлів

**Сценарій**: Файл `навчальний план (копія).md` — пробіли, дужки, кирилиця в назві.

**Захист**:
- Зберігати оригінальні імена файлів без змін
- Використовувати Path() замість string concatenation
- Тестувати на Windows, macOS, Linux

### P7.5: Case-sensitivity

**Сценарій**: `README.md` та `readme.md` — на Linux це різні файли, на Windows/macOS — один.

**Захист**:
- Використовувати точні імена файлів як в оригіналі
- При скануванні — перевіряти на дублікати з різним регістром, попереджувати

## Категорія 8: Git та контроль версій

### P8.1: Величезний diff

**Сценарій**: Перший переклад — 50 нових файлів, 200 KB. PR стає нечитабельним.

**Захист**:
- Рекомендація в документації: розбивати на окремі PR за пріоритетами
- Інструмент може генерувати коміт-повідомлення для кожної "пачки"
- `repo-translate run --batch P1` — тільки пріоритетні файли

### P8.2: Merge-конфлікти

**Сценарій**: Два перекладачі працюють з різних мов одночасно, обидва додають файли в translations/ → конфлікт.

**Захист**:
- Кожна мова — окремий підкаталог → конфліктів не буде (різні файли)
- `.repo-translator.yaml` — єдиний спільний файл, але він змінюється рідко

### P8.3: Squash merge втрачає проміжні коміти

**Сценарій**: Мейнтейнер робить squash merge PR → i18n-source-sha в метаданих не збігається з жодним комітом в main.

**Захист**:
- При sync-check — якщо SHA не знайдено, використовувати date-based пошук
- Або: зберігати hash файлу замість/на додаток до commit SHA

## Категорія 9: Конфігурація та налаштування

### P9.1: Невалідний YAML в конфігу

**Сценарій**: Користувач редагує .repo-translator.yaml і ламає синтаксис.

**Захист**:
- При завантаженні — валідація YAML + зрозумілі повідомлення помилок
- Fallback на дефолтні значення якщо конфіг зламаний (з ПОПЕРЕДЖЕННЯМ)

### P9.2: Невідомий код мови

**Сценарій**: `--lang ukr` замість `--lang uk`.

**Захист**:
- Валідація по ISO 639-1 / ISO 639-2
- Fuzzy match: "ukr" → "Можливо ви мали на увазі uk (Українська)?"

### P9.3: Глосарій з конфліктами

**Сценарій**: Глосарій має "deploy" → "розгортання" і "deployment" → "деплой" — несумісні переклади однокореневих слів.

**Захист**:
- При завантаженні глосарію — перевіряти на конфлікти (однокореневі слова з різними перекладами)
- ПОПЕРЕДЖЕННЯ з рекомендацією виправити

## Категорія 10: Крайні випадки

### P10.1: Файл що складається тільки з таблиць

**Сценарій**: Markdown файл = одна велика таблиця на 200 рядків. AI має перекладати вміст комірок, зберігаючи структуру.

**Захист**:
- Таблиці обробляти рядок-за-рядком (не як суцільний блок)
- Зберігати кількість `|` символів в кожному рядку
- Перевіряти після перекладу: кількість рядків таблиці = оригіналу

### P10.2: Markdown з HTML всередині

**Сценарій**: `<details><summary>Click to expand</summary>` — HTML теги в Markdown. AI може перекласти атрибути або зламати теги.

**Захист**:
- HTML-теги = плейсхолдери (як код-блоки)
- Текст між тегами — перекладати
- Атрибути тегів — НЕ перекладати

```html
<!-- Оригінал -->
<details>
<summary>Click to expand</summary>
Content here
</details>

<!-- Переклад -->
<details>
<summary>Натисніть щоб розгорнути</summary>
Вміст тут
</details>
```

### P10.3: LaTeX формули

**Сценарій**: `$E = mc^2$` або `$$\int_0^1 f(x)dx$$` — LaTeX в Markdown.

**Захист**:
- Виявляти LaTeX-блоки (`$...$` та `$$...$$`)
- Не перекладати (плейсхолдери)

### P10.4: Emoji в заголовках

**Сценарій**: `## 🚀 Getting Started` — emoji впливає на генерацію якорів.

**Захист**:
- При генерації якорів — видаляти emoji (як робить GitHub)
- `## 🚀 Getting Started` → `#-getting-started` (або `#getting-started`)

### P10.5: Файл без заголовків

**Сценарій**: Файл .md без жодного `#` заголовка — просто текст. Чанкування по заголовках неможливе.

**Захист**:
- Якщо немає `##` заголовків — чанкувати по порожніх рядках (параграфах)
- Якщо файл < 4000 токенів — перекладати цілком

### P10.6: Markdown-коментарі з метаданими

**Сценарій**: Файл вже має `<!-- i18n-source: ... -->` від попереднього запуску. Повторний запуск — додасть ще один заголовок.

**Захист**:
- Перед додаванням метаданих — перевірити чи вони вже є
- Якщо є — оновити (не дублювати)

### P10.7: Переклад перекладу

**Сценарій**: Користувач випадково запускає переклад на каталозі translations/ — інструмент перекладатиме вже перекладені файли.

**Захист**:
- ЗАВЖДИ виключати `translations_dir` зі сканування
- При скануванні — перевірити: чи source_dir ≠ target_dir
- Якщо користувач вказав source що знаходиться всередині translations/ — ПОМИЛКА

### P10.8: Рекурсивний include/exclude

**Сценарій**: Конфіг `include: ["**/*.md"]` а `exclude: ["CHANGELOG.md"]` — але CHANGELOG.md є в підкаталогах (`docs/CHANGELOG.md`). Exclude повинен працювати і для підкаталогів.

**Захист**:
- Використовувати glob-патерни з підтримкою `**/`
- `exclude: ["**/CHANGELOG.md"]` — виключає в будь-якому каталозі
- `exclude: ["CHANGELOG.md"]` — виключає тільки в корені

## Зведена таблиця захистів

### Автоматичні (без участі користувача):

| Захист | Коли спрацьовує | Дія |
|--------|----------------|-----|
| UTF-8 перевірка | Після кожного запису | Автовиправлення або відхилення |
| Код-блоки через плейсхолдери | При перекладі | Код ніколи не проходить через AI |
| Перевірка кількості рядків | Після перекладу | Відхилення при < 85% |
| Виправлення якорів | Після перекладу | Автоматична заміна |
| Перерахунок посилань | При створенні файлу | Автоматичне переписування шляхів |
| Newline в кінці файлу | При записі | Автоматичне додавання |
| Видалення invisible Unicode | Після перекладу | Автоматичне видалення |
| Перевірка дублювання метаданих | При додаванні заголовка | Оновлення замість дублювання |

### Попередження (потребують уваги користувача):

| Захист | Коли спрацьовує | Повідомлення |
|--------|----------------|-------------|
| Великий репозиторій | При скануванні | "12,847 файлів, рекомендуємо звузити" |
| Існуюча структура перекладів | При init | "Знайдено uk/, імпортувати?" |
| Невідповідність глосарію | При валідації | "hook перекладено як гачок (1x)" |
| RTL мова | При init | "Потрібна додаткова увага" |
| Абсолютні шляхи | При скануванні | "Не можуть бути переписані" |
| Squash merge SHA | При sync-check | "SHA не знайдено, використовую дату" |

### Блокуючі (зупиняють процес):

| Захист | Коли спрацьовує | Чому блокує |
|--------|----------------|-------------|
| Файл скорочено > 15% | Після перекладу | AI втратив контент |
| Код-блоки не збігаються | Після перекладу | Структура зламана |
| Source = Target каталог | При init | Переклад перекладу |
| Немає прав запису | При init | Неможливо зберегти |
| Диск заповнений | При записі | Неможливо продовжити |
