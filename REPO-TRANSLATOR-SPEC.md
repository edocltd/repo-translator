# repo-translator — Специфікація проєкту

## Аналіз ринку та позиціонування

### Існуючі рішення та їхні слабкості

| Інструмент | Підхід | Слабкості |
|-----------|--------|-----------|
| **Azure/co-op-translator** (596⭐) | CLI + GitHub Actions, Azure OpenAI / OpenAI, Python | Тільки Azure/OpenAI (не Anthropic). Немає валідації якорів. Немає глосарію на рівні проєкту. Немає human-in-the-loop. Не перевіряє структурну цілісність. Складне налаштування (Azure підписка) |
| **3ru/gpt-translate** (700⭐) | GitHub Action, коментар в issue → PR | Файл-за-файлом (не розуміє структуру репо). Без глосарію. Без валідації. Без порівняння з оригіналом |
| **rockbenben/md-translator** | Веб-інтерфейс | Тільки окремі файли, не репозиторій. Без git-інтеграції |
| **smikitky/chatgpt-md-translator** | CLI | Тільки один файл за раз. Без збереження структури. Без валідації |
| **Crowdin / Transifex / Lokalise** | SaaS | Орієнтовані на .json/.po/.xliff (software strings), не на markdown-документацію. Дорогі. Надлишкові для OSS |

### Наша ніша — що НІХТО не робить

1. **Контроль якості перекладу** — перевірка якорів, кількості рядків, збереження код-блоків, кодування UTF-8
2. **Проєктний глосарій** — один термін = один переклад по всьому репозиторію
3. **Класифікація файлів** — автоматичне визначення "перекладати / копіювати / пропустити"
4. **Підтримка Anthropic API** — ніхто не використовує Claude для перекладу репозиторіїв
5. **Human-in-the-loop** — AI перекладає, людина рев'ює, інструмент валідує

## Проблеми, виявлені при перекладі claude-howto

Це реальний досвід перекладу ~80 файлів / 400 KB документації. Кожна проблема — вимога до інструменту.

### P1: Класифікація файлів (що перекладати?)

**Проблема**: Репозиторій містить .md, .py, .sh, .json, .png, .svg, .yaml та інші файли. Потрібно знати для кожного: перекласти / скопіювати as-is / пропустити.

**Що ми робили вручну**: Порівнювали структуру оригіналу з vi/ та zh/ щоб зрозуміти що включати.

**Правила класифікації** (які потрібно автоматизувати):
- `.md` — перекладати (крім файлів що повністю складаються з коду/конфігу)
- `.txt` — перекладати якщо містить прозу
- `.py`, `.sh`, `.js`, `.json`, `.yaml` — копіювати as-is
- `.png`, `.svg`, `.jpg` — копіювати as-is
- `LICENSE`, `.gitignore`, `coverage.xml` — пропустити (використовувати з кореня)
- Frontmatter YAML у .md — НЕ перекладати `name`, `tools` поля; ПЕРЕКЛАДАТИ `description`

### P2: Збереження структури Markdown

**Проблема**: AI-модель "творчо скорочує" файли — вирізає приклади, код-блоки, діаграми. При claude-howto 4 з 10 модулів були скорочені на 18-58%.

**Як виявили**: Порівняння розмірів файлів (коректний переклад кирилицею = ~40% більший за оригінал, менший = скорочення).

**Вимога**: Після перекладу перевіряти:
- Кількість рядків (±5% від оригіналу, враховуючи мову)
- Кількість код-блоків (``` пар) — має бути ідентичною
- Кількість Mermaid-діаграм — ідентична
- Кількість таблиць — ідентична
- Код всередині блоків — побайтово ідентичний оригіналу

### P3: Якорні посилання

**Проблема**: При перекладі заголовків `## Slash Commands` → `## Слеш-команди` якорі змінюються (`#slash-commands` → `#слеш-команди`). Посилання у змісті стають зламаними.

**Як виявили**: Скрипт `check_cross_references.py` після перекладу показав 10 зламаних якорів у concepts guide + 3 в модулях.

**Вимога**: Після перекладу автоматично:
1. Знайти всі `[text](#anchor)` посилання
2. Порівняти з реальними заголовками (згенерувати GitHub-style якорі)
3. Автоматично виправити зламані якорі

### P4: Апостроф та Unicode

**Проблема**: В українській `'` (U+0027) та `ʼ` (U+02BC) — різні символи. GitHub anchor generator обробляє їх по-різному. Скрипт CC використав `ʼ` в посиланнях але `'` в заголовках → mismatch.

**Вимога**: Нормалізувати апострофи до одного варіанту у якорних посиланнях.

### P5: Кодування файлів

**Проблема**: CC записав частину файлу в CP1251 замість UTF-8. 240 невалідних байтів у файлі, невидимих при нормальному перегляді.

**Вимога**: Після кожного запису — перевірка валідності UTF-8. Якщо невалідний — спроба автовиправлення (визначення кодування блоку + перекодування).

### P6: Консистентність термінології

**Проблема**: "hook" у різних файлах перекладався як "хук", "гачок", "перехоплювач". Потрібен глосарій.

**Вимога**: Проєктний глосарій (glossary.json) що:
- Завантажується перед початком перекладу
- Передається в кожен AI-запит як контекст
- Перевіряється після перекладу (пошук неконсистентних перекладів)

### P7: Великі файли

**Проблема**: `claude_concepts_guide.md` — 87 KB / 3136 рядків. Не вміщується в один AI-запит. Потрібно розбивати на секції.

**Вимога**: Розбиття великих файлів по ## заголовках, переклад секція-за-секцією, збірка назад зі збереженням структури.

### P8: Відсутні файли

**Проблема**: При порівнянні uk/ з оригіналом виявились 6+ відсутніх файлів яких не помітили при ручному плануванні.

**Вимога**: Повний diff структури оригіналу vs перекладу з чітким звітом: що є, чого немає, що зайве.

### P9: Відносні посилання

**Проблема**: В перекладених файлах посилання типу `[Memory](../02-memory/)` повинні вказувати на каталоги ЗА МЕЖАМИ uk/ (на оригінал) або ВСЕРЕДИНІ uk/ (на переклад). Потрібна чітка стратегія.

**Вимога**: Налаштування — посилання йдуть на uk/ (якщо файл перекладений) або на оригінал (якщо ні).

### P10: Синхронізація з оригіналом

**Проблема**: Після мерджу оригінал буде оновлюватись. Потрібно відстежувати які файли застаріли.

**Вимога**: i18n-метадані в кожному файлі (source SHA, date) + скрипт що порівнює поточний SHA з записаним та генерує звіт застарілих файлів.

## Функціональні вимоги

### F1: Сканування репозиторію (`scan`)

**Вхід**: Шлях до репозиторію, цільова мова, конфігурація

**Вихід**: `translation-plan.json` — повний план що робити з кожним файлом

```json
{
  "source_repo": "/path/to/repo",
  "target_lang": "uk",
  "target_dir": "uk/",
  "files": [
    {
      "path": "README.md",
      "action": "translate",
      "type": "markdown",
      "size_bytes": 28728,
      "estimated_tokens": 7200,
      "priority": "P1",
      "depends_on": []
    },
    {
      "path": "scripts/build.py",
      "action": "copy",
      "type": "python",
      "reason": "code file, no prose"
    },
    {
      "path": ".gitignore",
      "action": "skip",
      "reason": "config, shared with root"
    },
    {
      "path": "01-slash-commands/pr-slash-command.png",
      "action": "copy",
      "type": "image"
    }
  ],
  "stats": {
    "total_files": 67,
    "to_translate": 42,
    "to_copy": 20,
    "to_skip": 5,
    "estimated_tokens": 180000,
    "estimated_cost_usd": 2.70
  }
}
```

**Логіка класифікації**:
1. За розширенням: `.md` → translate, `.py/.sh/.js` → copy, `.png/.svg` → copy
2. За вмістом: якщо .md файл на 90%+ код (YAML frontmatter + код-блоки) → copy
3. За конфігурацією: override правила для конкретних файлів/патернів
4. За існуючими локалізаціями: якщо інша мова вже є — дивитись що вони включили

### F2: Переклад файлу (`translate`)

**Вхід**: Оригінальний .md файл, глосарій, промпт-шаблон, цільова мова

**Процес**:
1. Класифікувати секції файлу: проза (перекладати) vs код (зберегти)
2. Якщо файл > 4000 токенів — розбити по ## заголовках
3. Для кожної секції: відправити AI-запит з глосарієм і правилами
4. Зібрати результат, додати i18n-метадані
5. Записати у UTF-8 з перевіркою кодування

**Правила для AI-промпту** (витягнуті з нашого ТЗ):
- Перекладати прозу, заголовки, описи в таблицях
- НЕ чіпати: код-блоки, Mermaid, JSON/YAML, шляхи, URL, CLI-команди
- Використовувати глосарій строго
- Зберігати кількість рядків ±5%
- Зберігати Markdown-форматування 1:1

### F3: Валідація перекладу (`validate`)

**Вхід**: Оригінальний файл + перекладений файл

**Перевірки**:

| # | Перевірка | Тип | Автовиправлення |
|---|-----------|-----|-----------------|
| V1 | UTF-8 валідність | Критична | Так (перекодування) |
| V2 | Кількість рядків ±5% | Попередження | Ні (потрібен ручний рев'ю) |
| V3 | Кількість код-блоків = оригінал | Критична | Ні |
| V4 | Кількість Mermaid-діаграм = оригінал | Критична | Ні |
| V5 | Код всередині блоків = оригінал | Критична | Так (копіювання з оригіналу) |
| V6 | Якорні посилання (#anchor) резолвляться | Критична | Так (fix-anchors.py) |
| V7 | Відносні .md посилання резолвляться | Критична | Ні |
| V8 | Глосарій-консистентність | Попередження | Ні |
| V9 | i18n-метадані присутні | Попередження | Так (додавання) |
| V10 | Trailing whitespace / EOL | Стиль | Так (trim) |

**Вивід**: Звіт JSON + людино-читабельний summary

```
=== Validation Report ===
✅ UTF-8 valid
✅ Lines: 3135 (original: 3135, delta: 0%)
✅ Code blocks: 194/194
✅ Mermaid diagrams: 31/31
⚠️  Glossary: 2 inconsistencies found
   - "hook" translated as "хук" (15x) and "гачок" (1x) in 06-hooks/README.md:245
❌ Broken anchors: 3
   - uk/04-subagents/README.md:#постійна-памʼять → should be #постійна-память
   [auto-fix available]
```

### F4: Виправлення якорів (`fix-anchors`)

**Вхід**: Перекладений .md файл

**Логіка**:
1. Витягти всі заголовки → згенерувати GitHub-style якорі
2. Знайти всі `[text](#anchor)` посилання
3. Якщо anchor не збігається — знайти найближчий валідний (fuzzy match)
4. Замінити зламані якорі на правильні
5. Нормалізувати апострофи (U+02BC → видалити з якорів, як робить GitHub)

### F5: Перевірка синхронізації (`sync-check`)

**Вхід**: Каталог перекладу (uk/), оригінальний репозиторій

**Логіка**:
1. Для кожного перекладеного файлу — прочитати `i18n-source-sha`
2. Отримати поточний SHA оригіналу (`git log --format=%h -1 -- <file>`)
3. Якщо SHA різні — файл застарів
4. Згенерувати diff оригіналу між старим та новим SHA
5. Вивести звіт: які файли потребують оновлення, що змінилося

**Вивід**:
```
=== Sync Report ===
Checked: 42 files
Up to date: 38 files
Outdated: 4 files

  README.md
    Source SHA: 63a1416 → a1b2c3d (changed 3 days ago)
    Changes: +15 -3 lines (new FAQ entry, updated stats)
    Priority: High (root file)

  06-hooks/README.md
    Source SHA: 63a1416 → f4e5d6c (changed 1 day ago)
    Changes: +42 -8 lines (new hook event added)
    Priority: Medium
```

### F6: Структурний diff (`diff-structure`)

**Вхід**: Оригінальний каталог, каталог перекладу

**Вивід**: Повне порівняння — що є, чого немає, що зайве

```
=== Structure Diff: original/ vs uk/ ===

MISSING in uk/ (12 files):
  ❌ claude_concepts_guide.md (87 KB) — translate
  ❌ resources/DESIGN-SYSTEM.md — translate
  ❌ 03-skills/code-review/scripts/analyze-metrics.py — copy

EXTRA in uk/ (2 files):
  ⚠️  uk/TRANSLATION_QUEUE.md — uk-specific, OK
  ⚠️  uk/TRANSLATION_NOTES.md — uk-specific, OK

MATCHING (55 files): ✅
```

## Взаємодія з користувачем

### Сценарій 1: Перший переклад репозиторію

```
Користувач: "Переклади цей репозиторій українською"

repo-translator:
1. [SCAN] Сканую репозиторій...
   Знайдено: 67 файлів
   → 42 для перекладу (.md)
   → 20 для копіювання (.py, .sh, .json, .png)
   → 5 пропустити (.git, .venv, LICENSE)
   Оцінка: ~180K токенів, ~$2.70

2. [GLOSSARY] Глосарій не знайдено.
   Створюю repo-translator-glossary-uk.json з дефолтними термінами.
   → Перевір та відредагуй перед стартом!

3. [PLAN] Створено translation-plan.json
   Пріоритети:
   P1: README.md, INDEX.md (ядро)
   P2: 10 модульних README
   P3: 42 файлів-прикладів
   P4: 5 допоміжних (CONTRIBUTING, etc.)

4. Підтвердити план? [Y/n]
```

## Метрики успіху

1. **Час перекладу**: Репозиторій розміру claude-howto (~80 файлів) за < 2 години (vs 3 дні вручну)

3. **Консистентність**: 100% відповідність глосарію
4. **Adoption**: 50+ зірок за перший місяць (якщо правильно позиціонувати)
